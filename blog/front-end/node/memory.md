# 内存与垃圾回收
## Node内存与整体架构
### 内存结构分配
V8下，**64位系统下大约为1.4G，32位系统下大约为0.7G**。
- 表层原因：是为JavaScript在浏览器打造的，不太可能遇到大量使用内存的场景，所以可以申请的最大内存就没设置太大
- 深层原因：是V8垃圾回收机制的限制
    - 以1.5G的堆内存为例，V8做一次小的垃圾回收需要50ms以上，做一次非增量式的垃圾回收需要1s以上。大内存分配必然导致性能的直线下降
    - 但也可以打开这种限制（单位kb），需要在程序执行前设置，没办法动态设置
    ``` sh
    <!-- 老生代 -->
    node --max-old-space-size=1700 app.js  

    <!-- 新生代 -->  
    node --max-new-space-size=1200 app.js
    ```

### 整体架构
Node.js主要分为四大部分，Node Standard Library，Node Bindings，V8，Libuv，
![node架构](/node/node-framework.jpg)
- **Node Standard Library**：node提供的标准库
- **Node Bindings**：js 和 c++ 的通讯桥梁，封装v8 和 libuv，向上提供API
- 底层：C++实现层
    - **v8**： JavaScript引擎，提供JavaScript运行环境
    - **libuv**：node封装库， 提供跨平台的异步IO能力；linux上使用epoll,OSX和BSD上使用kqueue,windows上使用IOCP。
    - C-ares：异步处理 DNS 的能力
    - http_parser、OpenSSL、zlib...：http解析、SSL、数据压缩等

## V8垃圾回收策略
### 1. 新生代
新生代内存分配：64位是32M，32位是16M。

V8的垃圾回收策略主要基于分代式垃圾回收机制。在GC的演变过程中，人们发现没有一种垃圾回收算法能够胜任所有场景。<br>
**V8中内存分为新生代和老生代**，新生代存活时间较短（一些变量），老生代存活时间较长(window、DOM、Web API等)。<br>
主垃圾回收器-Major GC负责老生代的垃圾回收 <br>
副垃圾回收器-Minor Gc负责新生代的垃圾回收，基于Scavenge算法
#### Scavenge 算法
新生代采用 Scavenge 垃圾回收算法，算法实现时主要采用cheney算法。<br>
cheney算法讲内存一分为二，叫做semispace，一块处于使用状态(from空间)，一块处于闲置状态(to空间)。

讲解一下scavenge流程
![scavenge算法](/node/scavenge.jpg)
1. from空间中声明变量 a、b，并调用了a变量。
2. GC进来判断 `var a; alert(a);` 存在调用，会将活跃变量 a 复制进入 to 空间。
3. 清空 from 空间中的全部内存，此时b被清除。
4. 交换 from 和 to 空间，开始下一轮。

由于Scavenge是典型的牺牲空间换取时间的算法，所以无法大规模的应用到所有的垃圾回收中。但我们可以看到，Scavenge非常适合应用在新生代中，因为新生代中对象的生命周期较短，恰恰适合这个算法。

#### 晋升
对象从新生代移动到老生代的过程叫做晋升

晋升条件
1. 如果一个对象是第二次经历from空间到to空间的复制，那么它将会被移动到老生代中。
2. 一个对象从from复制到to空间时，如果to空间已经使用了25%，那这个对象将移动到老生代中。

### 2. 老生代
老生代内存分配：64位是1.4G，32位是0.7G。

老生代中的对象存活时间较长，继续使用Scavenge算法会浪费一半的内存空间。<br>
V8老生代中主要使用 Mark-Sweep 和 Mark-Compact相结合的方式进行垃圾回收。
#### Mark-Sweep & Mark-Compact
- Mark-Sweep
    - **过程**： Mark-Sweep在标记阶段遍历堆内存中的所有对象，并标记活着的对象，在随后的阶段，只清除没有被标记的对象。
    - **缺点**： Mark-Sweep最大的问题就是进行一次垃圾回收之后，内存空间会出现不连续的情况。这种内存碎片会对后续内存分配造成问题。
- Mark-Compact
    - Mark-Compact是为了解决Mark-Sweep的内存碎片问题。
    - **过程**：Mark-Compact 在标记完存活对象以后，会将标记的对象向内存空间的一端移动，移动完成之后，直接清理掉边界以外的所有内存。

前两步是Mark-Sweep，后两步是Mark-Sweep + Mark-Compact
![mark](/node/mark-sweep&mark-compact.jpg)
由于Mark-Compact需要移动对象，它的执行速度就不可能很快。在取舍上，V8主要基于Mark-Sweep，在空间不足以对新生代中晋升过来的对象进行分配时才使用Mark-Compact。

## 垃圾回收机制
Scavenge、Mark-Sweep、Mark-Compact说明了V8中的内存是如何分配和回收的，那如何判断一个对象是可以被回收的呢？
### 1. 标记清除
1. 垃圾收集器在运行时会给存储在内存中的所有变量都加上标记
2. 然后去掉运行环境中的变量以及被运行环境变量所引用的变量的标记
3. 依然被标记的变量就是需要被回收删除的变量。垃圾收集器完成内存清除工作，销毁被标记的值并回收它们所占用的内存空间
### 2. 引用计数
- 当声明一个变量并将一个引用类型的值赋给该变量时，这个值的引用次数就是1
- 如果同一个值赋给另一个变量，那么该值的引用次数加1，相反，如果该值的引用变量改变了对象引用，该值的引用次数减1
- 垃圾回收器下次运行时，就会释放引用次数为0的的值所占用的内存
-------
![增量更新](/node/increment-mark.jpg)
左上常规标记，左下增量标记，右边是并行、并发

主线程停下来进行GC叫全停顿(Stop-The-World)，为了解决停顿带来的卡顿。V8内部还有并行、并发、增量等垃圾回收技术
### 3. 增量标记
垃圾回收器将标记工作分解为更小的块，并且穿插在主线程不同任务之间执行。

### 4. 并行
在执行一个完整的垃圾回收过程中，垃圾回收会使用多个辅助线程来并行执行垃圾回收。
### 5. 并发
回收线程在执行JavaScript的过程中，辅助线程能够在后台完成执行垃圾回收的操作。

## V8三色标记法
- 三色
    - 黑色：表示这个节点被GC Root引用到了，而且子节点都已标记完成
    - 白色：表示节点没有被访问到，如果在本轮遍历结束时还是白色，那么这块数据就会被回收
    - 灰色：表示这个节点被GC Root引用到，但子节点还没被垃圾回收器标记处理
- 写屏障
    - **写屏障机制会强制将被引用的白色节点变成灰色**。这样就保证了黑色节点不能指向白色节点的约束条件

![](/node/tricolor.jpg)
``` js
window.a = {}
window.a.b = {}
window.a.b.c = {}
// GC扫了一遍之后
window.a.b = []
```
1. 声明a、a.b、a.b.c
2. GC标记一遍以后
3. 执行 a.b = [],此时[]对应图中的d,GC未标记到，本应是白色，但写屏障将d直接标记为灰色，这样下次增量标记时可以直接从灰色节点开始标记。
